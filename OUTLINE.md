## 第四部分：服务端架构（SFU/MCU/录制）

### 第22章：MCU 混音混画详解 ⭐ 独立成章

> **目标**：深入理解 MCU 架构，实现音视频混合  
> **衔接**：解决 SFU 客户端压力大、布局不统一问题

**22.1 MCU 架构原理**
- 22.1.1 什么是 MCU？Multipoint Control Unit
- 22.1.2 MCU vs SFU 对比：
  - MCU：服务端混音混画，客户端只收一路
  - SFU：服务端转发，客户端收多路
- 22.1.3 MCU 适用场景：
  - 万人直播间（观众端只收一路）
  - 统一布局（所有观众看到相同画面）
  - 弱网观众（低带宽也能观看）
- 22.1.4 MCU 的挑战：
  - 服务端计算压力大（编解码）
  - 延迟增加（混音混画耗时）
  - 单点故障风险

**22.2 音频混音原理**
- 22.2.1 音频混音基础：PCM 数据相加
- 22.2.2 溢出处理：限幅、压缩
- 22.2.3 语音检测（VAD）：只混有声音的轨道
- 22.2.4 音量归一化：防止某一路声音过大
- 22.2.5 混音算法实现：
```cpp
// 简单混音（需溢出处理）
for (int i = 0; i < samples; i++) {
    int sum = pcm1[i] + pcm2[i] + pcm3[i];
    output[i] = Clamp(sum, -32768, 32767);
}
```

**22.3 视频混画原理**
- 22.3.1 视频布局设计：
  - 等分网格：2×2、3×3、4×4
  - 主讲人模式：大画面 + 小缩略图
  - 画廊模式：等大小平铺
- 22.3.2 YUV 图像缩放：libyuv、swscale
- 22.3.3 图像叠加：Alpha 混合、位置计算
- 22.3.4 GPU 加速混画：OpenGL、CUDA、Metal

**22.4 MCU 流水线设计**
- 22.4.1 输入处理：接收多路 RTP，解码为 YUV/PCM
- 22.4.2 混音模块：多路 PCM → 一路 PCM
- 22.4.3 混画模块：多路 YUV → 一路 YUV
- 22.4.4 编码输出：混音混画后重新编码
- 22.4.5 延迟分析：解码 + 混音 + 混画 + 编码 < 100ms

**22.5 开源 MCU 分析**
- 22.5.1 mediasoup：SFU 为主，支持 pipeToRouter
- 22.5.2 Licode：早期 MCU 方案
- 22.5.3 Jitsi Videobridge：SFU + 可选 MCU
- 22.5.4 Janus：插件化，支持 MCU 插件

**22.6 简单 MCU 实现**
- 22.6.1 架构设计：输入管理器 + 混音器 + 混画器 + 编码器
- 22.6.2 音频混音实现（软件）
- 22.6.3 视频混画实现（OpenGL 加速）
- 22.6.4 性能优化：多线程、GPU 加速

**22.7 MCU 性能调优**
- 22.7.1 CPU 优化：SIMD 指令（SSE/AVX/NEON）
- 22.7.2 GPU 优化：CUDA、Metal、OpenCL
- 22.7.3 布局缓存：静态部分不重复渲染

### 第24章：WebTransport 与 WHEP/WHIP ⭐ 最新技术

> **目标**：探索最新 Web 传输技术  
> **衔接**：RTMP/WebRTC 的替代方案

**24.1 WebTransport 协议**
- 24.1.1 为什么需要 WebTransport？
  - WebRTC DataChannel 复杂性
  - TCP 队头阻塞
- 24.1.2 WebTransport 特性：
  - 基于 HTTP/3 + QUIC
  - 可靠流传输（类似 TCP）
  - 不可靠数据报（类似 UDP）
- 24.1.3 浏览器支持：Chrome 97+
- 24.1.4 服务端实现： aioquic、quiche

**24.2 WHIP 协议（WebRTC-HTTP Ingestion Protocol）**
- 24.2.1 WHIP 简介：简化 WebRTC 推流
- 24.2.2 信令简化：HTTP POST 代替 WebSocket
- 24.2.3 与 RTMP 对比：
  - 延迟：WHIP < 500ms，RTMP 1-3s
  - 浏览器支持：WHIP 直接支持，RTMP 需要 Flash
- 24.2.4 WHIP 服务端实现

**24.3 WHEP 协议（WebRTC-HTTP Egress Protocol）**
- 24.3.1 WHEP 简介：简化 WebRTC 播放
- 24.3.2 使用场景：浏览器播放 WebRTC 流
- 24.3.3 与 HLS/DASH 对比：延迟优势
- 24.3.4 WHEP 播放器实现

**24.4 SRT 协议**
- 24.4.1 SRT 简介：Secure Reliable Transport
- 24.4.2 ARQ 重传机制：精准重传
- 24.4.3 适用场景：跨地域直播、广播级传输
- 24.4.4 FFmpeg 支持：srt:// 协议

**24.5 协议选型指南**
| 场景 | 推荐协议 | 延迟 | 特点 |
|:---|:---|:---:|:---|
| 浏览器推流 | WHIP | <500ms | 原生 WebRTC，无需插件 |
| 浏览器播放 | WHEP | <500ms | 低延迟，优于 HLS |
| 跨地域传输 | SRT | 1-3s | 抗丢包，广播级质量 |
| 传统直播 | RTMP | 1-3s | 生态完善，CDN 支持好 |
| 实时连麦 | WebRTC | <300ms | 完整栈，P2P/SFU 支持 |
